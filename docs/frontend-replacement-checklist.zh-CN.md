# 前端替换清单（中文）

本文用于回答一个直接的问题：现在的新前端，到底替换了旧前端多少，还剩什么没有彻底替换。

## 1. 当前总体判断

当前仓库已经进入“新前端主流程可用，旧前端保留少量兼容层”的阶段。

更准确地说：

- 数据库管理、BLAST 提交、任务中心、结果主浏览已经迁移到新前端
- 新前端已是当前开发和联调的主线
- 旧前端仍保留少量历史页面和兼容接口

## 2. 新前端已覆盖内容

### 页面覆盖

已完成页面：

- `/`
- `/databases`
- `/blast/new`
- `/jobs`
- `/jobs/blast/:id`
- `/jobs/database/:id`

### 功能覆盖

已由新前端承担的主能力包括：

- 数据库列表
- 多来源导入
- 数据库索引任务查看
- 删除数据库
- BLAST 提交
- 基于历史任务回填
- 任务列表与状态查看
- BLAST 结果 query 导航
- hit table
- pairwise alignment 浏览
- 序列预览与 FASTA 下载
- 结果下载
- 图形总览与 alignment 图形浏览

## 3. 当前仍保留的旧前端 / 旧兼容层

从后端兼容逻辑看，当前仍保留：

- 旧搜索页相关页面与接口
- 旧结果页相关页面与接口
- `/:jid.json`
- `/searchdata.json`
- `/get_sequence`
- `/download/:jid.:type`
- `POST /cloud_share`

这些能力当前的定位已经不是主入口，而是兼容层。

## 4. 本地开发时的实际入口

当前本地开发模式默认启用 `api_only: true`，因此：

- 页面入口：`http://127.0.0.1:5174/`
- API：`http://127.0.0.1:4567/`
- Swagger UI：`http://127.0.0.1:4567/api/docs`

这说明在开发模式下，旧后端页面已经不再承担默认页面入口。

## 5. 替换状态判断

### 已经完成替换

- 数据库管理主流程
- BLAST 提交主流程
- 任务中心主流程
- 结果主浏览主流程
- 本地开发默认入口

### 已经基本替换，但仍保留兼容层

- 旧结果 URL `/:jid`
- 旧下载接口
- 旧序列提取接口
- 旧搜索数据接口

### 尚未完全收口

- 旧搜索页相关兼容策略
- `cloud_share` 的最终去留
- 少量旧结果页高级交互的完全替代

## 6. 当前最合理的结论

如果问题是“新前端能不能作为现在的主前端使用”，答案是：可以。

如果问题是“旧前端是不是已经可以完全删除”，答案是：还不能。

当前更准确的判断是：

- 新前端已经承担主线
- 旧前端已降级为兼容层
- 下一阶段重点是收口遗留入口，而不是重新设计主流程

## 7. 下一步建议顺序

建议继续按这个顺序推进：

1. 继续打磨 BLAST 结果页的排版和高级交互
2. 明确旧接口的长期保留 / 下线策略
3. 根据生产部署方式决定是否长期保持 `api_only`
4. 在版本说明中明确“主线前端已经切换，新旧兼容层尚未完全移除”
