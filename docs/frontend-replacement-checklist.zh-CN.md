# 新前端替代旧前端清单（中文）

本文用于回答一个非常具体的问题：

- 旧前端当前到底还承担哪些页面、入口和交互
- 新前端已经覆盖了哪些
- 还有哪些没有彻底替代
- 接下来应该按什么顺序把旧前端退下去

这份清单的目标不是描述“理想架构”，而是为实际替代工作提供一张可执行对照表。

## 1. 当前总体判断

当前仓库已经进入“新前端主流程可用、旧前端仍承担少量历史入口”的阶段。

可以简单理解为：

- 后端 API 主流程已经成型
- 新前端已经覆盖数据库管理、BLAST 提交、任务中心、任务详情、结果主浏览
- 旧前端仍然保留少量历史兼容流程
- 某些旧接口虽然还在，但已经不再适合作为主入口

截至当前版本，更准确的状态不是“新前端刚起步”，而是：

- 主流程已经迁移到新前端
- 剩余工作重点变成“替代遗留入口”和“收口部署方式”

## 2. 旧前端实际承担的路由和页面

从后端路由和 `views/*.erb` 来看，旧前端仍然直接承担以下页面或入口：

### 2.1 搜索页

- 页面路由：`GET /`
- 模板：`views/search.erb`
- 配套数据：`GET /searchdata.json`
- 提交搜索：`POST /`

这套流程本质上仍然是：

- Sinatra 返回 HTML
- 旧前端脚本读取 `/searchdata.json`
- 表单提交仍走旧后端页面模式

### 2.2 旧报告页

- 页面路由：`GET /:jid`
- 数据接口：`GET /:jid.json`
- 模板：`views/report.erb`

这意味着：

- 旧结果 URL 仍然保留
- 当新前端构建产物存在时，`GET /:jid` 已跳转到 `/jobs/blast/:id`
- 只有在新前端构建不可用时，才回退到旧 `views/report.erb`

### 2.3 旧结果辅助接口

- `GET /get_sequence/`
- `POST /get_sequence`
- `GET /download/:jid.:type`
- `POST /cloud_share`

这些接口原本主要服务于旧报告页及其侧边栏、弹窗和导出逻辑。

## 4. 替代状态清单

下面这张表是当前最有用的替代判断表。

### 4.1 页面和入口维度

#### 旧搜索页 `GET /`

- 当前状态：已被默认入口替代
- 新前端对应：`/blast/new`
- 评价：后端默认首页已切到新前端 SPA，旧搜索页不再是默认入口
- 结论：主入口已替代，旧搜索相关接口仍在兼容层保留

#### 旧报告页 `GET /:jid`

- 当前状态：主入口已替代
- 新前端对应：`/jobs/blast/:id`
- 评价：旧 URL 仍然存在，但默认行为已经迁移到新前端结果页
- 结论：主入口已替代，仍保留兼容入口

#### 旧搜索数据接口 `/searchdata.json`

- 当前状态：部分替代
- 新前端对应：`GET /api/v1/frontend/blast_form`
- 评价：新前端已开始使用新的表单配置接口，但旧搜索页仍依赖旧接口
- 结论：短期保留，长期可降级为兼容层

#### 旧提交搜索 `POST /`

- 当前状态：未替代
- 新前端对应：`POST /api/v1/blast_jobs`
- 评价：新旧两套提交方式并存
- 结论：仍需替代

#### 旧结果数据 `GET /:jid.json`

- 当前状态：部分替代
- 新前端对应：`GET /api/v1/blast_jobs/:id/result`
- 评价：新前端已不依赖它，但旧报告页仍依赖它
- 结论：短期保留，长期可降级为兼容层

#### 旧序列提取 `/get_sequence/` 和 `/get_sequence`

- 当前状态：部分替代
- 新前端对应：
  - `GET /api/v1/sequences`
  - `GET /api/v1/sequences/download`
  - `POST /api/v1/sequences/download`
- 评价：新前端已有新接口；旧接口主要为旧页面兼容
- 结论：主流程已替代，兼容层未移除

#### 旧下载 `/download/:jid.:type`

- 当前状态：部分替代
- 新前端对应：
  - `GET /api/v1/blast_jobs/:id/download/:type`
- 评价：新前端已切到新下载接口，但旧接口还在
- 结论：主流程已替代，兼容层未移除

#### 旧 cloud share 前端入口

- 当前状态：已隐藏
- 新前端对应：无
- 评价：后端保留，前端不再暴露
- 结论：当前按“隐藏待改造”处理

## 5. 现在还没有彻底完成替代的关键点

如果目标是“让新前端完全代替旧前端”，当前真正剩下的重点已经比较集中。

### 5.1 默认入口替代

这一项已经完成：

- `GET /` 已优先返回 `sequenceserver-web/dist/index.html`
- `/databases`
- `/blast/new`
- `/jobs`
- `/jobs/blast/:id`
- `/jobs/database/:id`

都已经由后端直接托管到新的前端 SPA 入口。

### 5.2 旧结果页路由替代

这一项已经部分完成：

- `GET /:jid`

现在默认会跳转到：

- `/jobs/blast/:id`

这意味着：

- 新前端结果页已经成为默认结果入口
- 旧结果 URL 仍作为兼容 URL 保留
- 旧模板只在新前端构建不可用时作为兜底

### 5.3 部署切换策略尚未完全收口

这一步已经部分完成：

- 方案 B 已经落地一半：Sinatra 现在会直接托管新前端构建产物
- 新前端默认入口和主页面路由已经可以由后端直接承载

但还没有完全收口的部分是：

- 旧 `/:jid` 仍作为兼容 URL 存在
- 旧兼容接口仍然保留
- 是否长期保留旧搜索相关接口还没有最终决定

## 6. 建议的收口顺序

下一阶段建议不要再平均发力，而是按下面顺序收口。

### 第一步：继续处理旧结果页迁移

默认首页切换和旧结果页默认跳转都已经完成，下一步最应该优先解决：

- 是否长期保留 `/:jid` 作为兼容入口
- 是否进一步让旧提交相关流程只保留 API 主路径

### 第二步：决定兼容入口保留时长

建议明确以下策略之一：

- 保留 `/:jid` 跳转一段时间后再移除
- 长期保留 `/:jid` 作为兼容 URL
- 在部署层限制旧兼容 URL 的对外暴露方式

### 第三步：把旧接口降级为兼容层

包括：

- `/:jid.json`
- `/get_sequence`
- `/download/:jid.:type`

目标不是立刻删掉，而是明确：

- 新前端主流程不再依赖它们
- 它们只是历史兼容接口

### 第四步：决定是否长期保留旧搜索页

如果不打算长期保留，后面就应该让：

- `GET /`
- `POST /`
- `/searchdata.json`

逐步退出主流程。

## 7. 当前最接近“完全替代”的定义

如果按工程目标来定义，“新前端完全代替旧前端”不一定要求你马上删除所有旧代码。

更现实的定义应该是：

- 用户默认访问的是新前端
- 新搜索、新结果、新数据库管理、新任务中心都走新前端
- 旧页面只作为兼容入口保留
- 新前端不再依赖旧页面专用接口

按照这个标准，当前已经非常接近完成，真正剩下的关键收口点主要是：

- 兼容入口保留策略
- 兼容接口收口
- 部署说明与长期保留策略整理

## 8. 下一步建议

最推荐的下一步不是再补一个零散小功能，而是直接推进：

- “旧结果页与兼容接口收口方案”

也就是明确：

1. 旧 `/:jid` 要保留多久作为兼容入口
2. `/:jid.json`、`/get_sequence`、`/download/:jid.:type` 是否进入兼容模式
3. 文档和部署默认值是否全部以新前端为主

首页切换完成后，项目已经进入“新前端主导、旧前端兼容”的阶段。
